🚀 Primeros pasos en NixOS

¿Qué es NixOS? Es una distribución de Linux donde todo se configura desde un archivo: servicios, paquetes, usuarios, red, bootloader... No usas comandos como apt install ni modificas archivos a mano. Escribes tu configuración y NixOS la hace realidad.

Esto se llama modelo declarativo: escribes lo que quieres y NixOS se encarga de traerte lo que pediste. Es como escribir la carta a los Reyes Magos: solo escribes tus deseos y alguien se encarga de traerte todo.

¿Qué es nixpkgs? Es el repositorio gigante con miles de programas para Nix/NixOS. Cuando dices pkgs.firefox, está diciendo "del repositorio nixpkgs, tráeme Firefox". Puedes buscar paquetes en search.nixos.org/packages.

Conceptos fundamentales

El Nix Store — /nix/storeEs la carpeta donde Nix guarda TODOS los archivos del sistema: programas, librerías, configuraciones. Cada versión de cada paquete tiene su propia carpeta con un hash único.
Donde Nix guarda todo. Cada paquete/version tiene su carpeta única.
# Ejemplo:
/nix/store/abc123-firefox-120.0/      # Firefox
/nix/store/def456-glibc-2.38/        # Librería C
configuration.nix — tu configuraciónEl archivo principal. Aquí defines TODO: paquetes, servicios, usuarios, red, bootloader. Está en /etc/nixos/configuration.nix
El archivo principal del sistema. Editas → reconstruyes → listo.
# Editar:
sudo vim /etc/nixos/configuration.nix
# Aplicar:
sudo nixos-rebuild switch
pkgs — la colección de paquetesEs la variable que contiene todos los paquetes disponibles. Cuando escribes pkgs.firefox, dices "del repositorio, quiero firefox".
pkgs es una variable con miles de programas.
environment.systemPackages = with pkgs; [ firefox git vlc ];
with pkgs; es un atajo: en lugar de escribir pkgs.firefox pkgs.git, solo escribes firefox git
Perfil y generaciones — tu sistema versionadoUn perfil apunta a la generación actual. Cada rebuild crea una nueva generación. Podés volver atrás (rollback).
Cada rebuild crea una "generación". Podés volver atrás si algo falla.
# Ver generaciones:
sudo nix-env -p /nix/var/nix/profiles/system --list-generations
# Volver atrás:
sudo nixos-rebuild switch --rollback

El flujo de trabajo

  • Editas tu configuración
    sudo vim /etc/nixos/configuration.nix
  • Reconstruyes el sistema
    sudo nixos-rebuild switch
  • Listo — tu sistema refleja la config
    Si algo falla: --rollback te salva

    • Editor recomendado: Usa vim si empiezas. Ejemplo: sudo vim /etc/nixos/configuration.nix. Para salir: Esc, luego :wq, luego Enter.

    💡 Dato importante: El archivo /etc/nixos/hardware-configuration.nix se genera automáticamente y contiene la config de tu hardware (discos, particiones, etc.). No lo edites a mano, se regenera cada vez que ejecutas nixos-generate-config. Tu config principal solo necesita tener imports = [ ./hardware-configuration.nix ]; al principio.

    🧹 Garbage Collector

    ¿Qué es el Garbage Collector? En NixOS nada se borra automáticamente. Cada vez que actualizas el sistema, se crea una nueva generación (una foto completa de tu sistema) y todos los archivos se guardan en /nix/store. Con el tiempo esto acumula gigas de versiones que ya no usas.

    El GC es el comando que libera ese espacio. Primero borras generaciones (los "punteros") y luego el GC elimina los archivos reales que ya nadie referencia. Es como vaciar la papelera de reciclaje.

    La generación actual (la que estás usando ahora) NUNCA se elimina. Siempre puedes hacer rollback.

    IMPORTANTE: No tener compilaciones o descargas activasAplica a: nix-collect-garbage, -d y --delete-older-than. Si nixos-rebuild está descargando/compilando y el GC borra esas referencias, falla con "path does not exist". al ejecutar el GC.

    RECOMENDADO
    GC conservador (30 días)Borra generaciones con +30 días de TODOS los perfiles y ejecuta GC. Mantiene la generación actual y las de los últimos 30 días. Ideal para uso semanal sin perder rollback.Ej: rebuild de hace 35 días ▶ se borra
    Generaciones viejas + espacio liberado en un paso.
    sudo nix-collect-garbage --delete-older-than 30d
    MODERADO
    GC simpleSolo elimina archivos del store sin referencias. No borra generaciones. Útil cuando ya borraste generaciones a mano y quieres liberar el espacio.Ej: tras borrar generaciones manualmente, libera GB
    Libera espacio sin borrar generaciones.
    sudo nix-collect-garbage
    AGRESIVO
    Limpieza profunda (-d)Borra TODAS las generaciones antiguas de TODOS los perfiles + GC. Pierdes rollback a versiones antiguas. Úsalo solo cuando estés seguro de que todo funciona.Ej: tras confirmar que todo funciona bien
    Elimina generaciones de todos los perfiles + GC.
    sudo nix-collect-garbage -d
    SEGURO
    Optimizar storeDeduplica archivos idénticos usando hardlinks. No borra nada, solo optimiza el espacio. Puede tardar un poco pero es seguro.Ej: 3 versiones de glibc ▶ ocupa una
    Deduplica archivos (no borra, solo optimiza).
    sudo nix-store --optimise

    ¿Cuál usar?

    Limpieza semanal--delete-older-than 30dSEGURO
    Tras borrar generaciones a manonix-collect-garbageSEGURO
    Limpieza profunda-dAGRESIVO
    Espacio extra sin borrarnix-store --optimiseSEGURO

    🗑️ Gestionar generaciones

    ¿Qué es una generación? Cada vez que ejecutas nixos-rebuild switch, NixOS crea una nueva generación: una "foto" completa de tu sistema en ese momento con todos los paquetes, configuraciones y versiones exactas.

    Esto te permite volver atrás si algo se rompe. Pero cada generación ocupa espacio. Gestionar generaciones significa decidir cuántas conservar y borrar las que ya no necesitas.

    Generaciones ≠ espacio liberado. Borrar generaciones solo quita los "punteros". Para liberar GB reales necesitas ejecutar el GC (pestaña GC).

    Ver generaciones del sistemaMuestra todas las generaciones del perfil system con su número, fecha y si es la actual (current). Útil para saber cuántas tienes y cuáles borrar.
    </>bash
    sudo nix-env -p /nix/var/nix/profiles/system --list-generations
    Eliminar excepto las últimas 5Mantiene las 5 generaciones más recientes del perfil system. Borra el resto. Ideal si haces rebuild frecuentemente y quieres un balance entre espacio y rollback.Ej: 20 generaciones ▶ borra 15
    </>bash
    sudo nix-env -p /nix/var/nix/profiles/system --delete-generations +5
    Eliminar las de más de 30 díasBorra generaciones del perfil system con +30 días. Más conservador que +5.
    </>bash
    sudo nix-env -p /nix/var/nix/profiles/system --delete-generations 30d

    Riesgo de --delete-generations

    NivelComandoDescripción
    SEGURO30dConserva el último mes
    MODERADO+5Conserva las 5 últimas
    AGRESIVOoldSolo deja la actual

    Para los comandos de GC (que liberan espacio real), consulta la pestaña GC arriba.

    🔧 Limpiar bootloader

    ¿Qué es el bootloader? Es el menú que ves al encender el ordenador donde eliges qué generación de NixOS arrancar. Cada generación crea una entrada en ese menú.

    Cuando borras generaciones, los archivos de esas entradas desaparecen pero las entradas en el menú pueden quedar huérfanas. Esta sección es para limpiarlas.

    ⚠️ ADVERTENCIA — Plasma Desktop

    Limpiar generaciones + GC puede romper los iconos anclados de Plasma.

    Los launchers guardan rutas absolutas del Nix Store. Al ejecutar GC, esas rutas desaparecen.

    • Afecta a TODAS las apps ancladas (Discord, Telegram, KeePassXC...)
    • La solución: editar ~/.config/plasma-org.kde.plasma.desktop-appletsrc y usar applications:nombre.desktop

    Orden seguro:

    1. sudo nixos-rebuild switch
    2. sudo nix-collect-garbage -d
    3. sudo nixos-rebuild boot

    ✓ Recomendado
    Opción B — Regenerar bootloaderEjecuta nixos-rebuild boot para reconstruir el menú de arranque desde cero. Solo aparecen las generaciones válidas. No deja entradas fantasma.Ej: 5 generaciones válidas ▶ el menú solo muestra esas 5
    Reconstruye el menú desde cero con solo las generaciones vigentes.
    </>bash
    sudo nixos-rebuild boot

    Las entradas reported/absent en bootctl list son variables EFI inofensivas que se borran al reiniciar. nixos-rebuild boot reconstruye el menú desde cero con solo las generaciones válidas.

    ❄️ Flakes

    ¿Qué son los Flakes? Son la forma moderna de gestionar tu configuración de NixOS. Imagina que tu configuración es un "proyecto" con sus propias dependencias versionadas, como un proyecto de programación con su package.json.

    El archivo flake.nix declara qué usar y el flake.lock "congela" las versiones exactas para que la compilación sea siempre idéntica. Esto hace que tu sistema sea reproducible: si copias los archivos a otro ordenador, obtienes exactamente lo mismo.

    Canales vs Flakes: Los canales son como un "apt update" global (simple pero menos control). Los flakes son proyectos independientes con dependencias versionadas (más control, más reproducible).

    ¿Uso Flakes o Canales?Comando para verificar tu configuración actual. Si usa canales (nixos-channel), cada rebuild puede cambiar versiones. Si usa flakes, las versiones están "congeladas" en flake.lock.nix-channel --list ▶ muestra canales activos
    </>bash
    nix-channel --list

    Si muestra nixos https://nixos.org/channels/... ▶ Canales. Si no ▶ Flakes.

    Estructura típicaOrganización de un proyecto con flakes. flake.nix declara qué dependencias usar (nixpkgs, home-manager). flake.lock guarda las versiones exactas (como package-lock.json). nixos/ contiene la configuración del sistema.├── flake.nix ▶ define inputs y outputs ├── flake.lock ▶ versiones exactas └── nixos/ ▶ tu config
    </>tree
    mi-config/
├── flake.nix        # Dependencias y outputs
├── flake.lock       # Versiones congeladas
└── nixos/
    ├── configuration.nix
    └── hardware-configuration.nix
    Actualizar dependenciasEquivalente a nix-channel --update pero con flakes.
    </>bash
    nix flake update
    Actualizar solo un inputActualiza un input específico sin tocar los demás. Útil cuando solo necesitas actualizar una dependencia (ej: home-manager) sin riesgo de romper otras.nix flake update home-manager
    </>bash
    nix flake update nixpkgs

    🔀 Configuración Híbrida

    ¿Qué es una config híbrida? Es un punto medio entre canales y flakes. Usas canales para el sistema base pero importas paquetes inestables con fetchTarball sin necesidad de flakes. Es lo que usa tu config actual.

    ¿Por qué? Algunos paquetes (Neovim, Brave, Git, etc.) se actualizan más rápido en unstable. Con este método tienes un sistema estable pero apps siempre al día.

    Importar nixpkgs-unstableCrea una variable "unstable" que contiene nixpkgs de la rama inestable. Se descarga en cada rebuild (no está congelado). Permite usar paquetes más nuevos sin cambiar todo el sistema.let unstable = import (fetchTarball "https://...") {};
    </>nix
    let
  unstable = import (fetchTarball "https://nixos.org/channels/nixpkgs-unstable/nixexprs.tar.xz") {
      config.allowUnfree = true;
  };
in
    Usar paquetes unstable en user packagesPuedes mezclar paquetes de ambas ramas. pkgs.* usa la estable, unstable.* usa la inestable. Ideal para tener el sistema estable pero apps actualizadas (Neovim, Brave, Git).pkgs.firefox (estable) + unstable.neovim (nuevo)
    </>nix
    users.users.zebus3d = {
    packages = with pkgs; [
        kdePackages.kate          # ◀ estable
        unstable.neovim           # ◀ inestable
        unstable.brave            # ◀ inestable
        unstable.git              # ◀ inestable
        direnv                     # ◀ estable
    ];
};

    fetchTarball vs flakes: fetchTarball descarga cada vezCada nixos-rebuild verifica si hay nueva versión. Más lento pero siempre actual. Flakes usa flake.lock para versiones fijas. — no congela versiones. Si quieres reproducibilidad total, migra a flakes.

    🔄 Actualizar el sistema

    ¿Qué significa actualizar en NixOS? A diferencia de otras distros donde "actualizar paquetes" es un comando separado, aquí actualizar significa: (1) descargar las últimas versiones del canal o flake, y (2) reconstruir todo el sistema con esas versiones. Kernel, apps, librerías — todo se actualiza de golpe.

    ¿Cada cuánto? Cada 1-2 semanas es lo ideal. Diario genera demasiadas generaciones. Esperar meses aumenta el riesgo de romper algo. Usuario casual ▶ una vez al mes.

    Canales

    Actualizar y aplicar en un pasoCombina nix-channel --update + nixos-rebuild switch en un solo comando. Actualiza el canal y reconstruye el sistema inmediatamente. Rápido pero no permite revisar qué cambió.sudo nixos-rebuild switch --upgrade
    </>bash
    sudo nixos-rebuild switch --upgrade
    En dos pasos (revisar antes)Útil si quieres ver qué cambió antes de aplicar.
    </>bash
    sudo nix-channel --update
sudo nixos-rebuild switch

    Flakes

    1. Actualizar flake.lock
    </>bash
    nix flake update
    2. Aplicar configuración
    </>bash
    sudo nixos-rebuild switch --flake .

    Con host específico:

    </>bash
    sudo nixos-rebuild switch --flake .#nombre-de-host

    Config Híbrida (fetchTarball)

    si usas el enfoque híbrido como tu config actual, no necesitas actualizar nada manualmente. fetchTarball descarga la última versión de nixpkgs-unstable en cada nixos-rebuild.

    Reconstruir con config híbridaEl fetchTarball se actualiza solo al hacer rebuild. No hay paso extra.
    </>bash
    sudo nixos-rebuild switch

    FetchTarball vs flakes: fetchTarball siempre descarga lo últimoCada rebuild verifica si hay nueva versión de nixpkgs-unstable. Más lento pero siempre actual. Flakes usa flake.lock para versiones fijas y es más rápido. — cada rebuild descarga. Si quieres velocidad, migra a flakes.

    📦 Instalar y buscar paquetes

    ¿Cómo se instala software en NixOS? Hay dos formas: declarativa (añades el paquete a configuration.nix y se instala al hacer rebuild) o imperativa (lo instalas al momento con nix-env). La declarativa es la recomendada porque es reproducible y persistente.

    📜 La carta a los Reyes Magos — La configuración declarativa es como escribir la carta a los Reyes Magos: escribes todos tus deseos en un papel (el archivo configuration.nix), describes exactamente qué quieres tener en tu sistema, y cuando ejecutas nixos-rebuild switch "llegan los Reyes" y te traen todos los regalos de golpe. Si quieres algo nuevo, lo añades a la carta y listo. Así de fácil: escribes lo que quieres y ¡voilà!, ya lo tienes.

    ¿Dónde está mi configuración? El archivo principal es /etc/nixos/configuration.nix. Ahí se define TODO: paquetes, servicios, usuarios, red, etc.

    search.nixos.org/packages
    Buscar desde la terminalBusca paquetes sin salir de la terminal. Muestra nombre, versión y descripción. Requiere tener nix-command activado en experimental-features.nix search nixpkgs blender
    Busca paquetes sin abrir el navegador:
    </>bash

    ⚙️ Automatización

    ¿Por qué automatizar? NixOS acumula espacio y generaciones con cada actualización. En lugar de limpiar manualmente, puedes configurar timers que lo hagan solos. Todo se configura en /etc/nixos/configuration.nix.

    Limpieza automática semanalTimer de systemd que ejecuta GC automáticamente. "weekly" = una vez por semana.Ej: cada domingo limpia generaciones viejas
    </>nix
    nix.gc = {
    automatic = true;
    dates = "weekly";
    options = "--delete-older-than 30d";
};
    Optimización automática del storeDeduplica archivos idénticos automáticamente al añadir algo al store.
    </>nix
    nix.settings.auto-optimise-store = true;
    Limitar generaciones en el menú de arranqueSystemd-boot solo muestra las N generaciones más recientes. No borra, solo oculta del menú.
    </>nix
    boot.loader.systemd-boot.configurationLimit = 10;

    🧩 Extras

    Herramientas adicionales para desarrolladores y usuarios avanzados: nix-shell para entornos aislados, nix-ld y wrappers para binarios externos, y más.

    nix-shell — Entornos aislados

    ¿Qué es nix-shell? Crea un entorno temporal con las dependencias que necesites. Es como un virtualenv de Python pero para cualquier lenguaje o herramienta. Dentro del shell tienes acceso a esos paquetes; afuera, no existen.

    Es ideal para: no ensuciar el sistema con dependencias de un proyecto, probar herramientas rápido, o tener versiones distintas de un mismo paquete según el proyecto.

    Shell interactivo con paquetesAbre un shell temporal donde los paquetes especificados están disponibles. Al salir (exit), el entorno desaparece. Ideal para probar algo rápido sin ensuciar el sistema.nix-shell -p python3 nodejs gcc
    Entra a un shell con las herramientas que necesites:
    </>bash
    nix-shell -p python3 nodejs gcc

    Dentro del shell puedes usar python3, node, gcc. Al escribir exit, desaparecen.

    Ejecutar un comando y salirEjecuta un comando con los paquetes especificados y sale automáticamente. Sin entrar a un shell interactivo. Útil para scripts o pruebas rápidas.nix-shell -p python3 --run "python --version"
    </>bash
    nix-shell -p python3 --run "python --version"
    ✓ Recomendado
    shell.nix — Entorno reproducibleCrea un archivo shell.nix en tu proyecto. Cualquiera que ejecute "nix-shell" obtiene el mismo entorno.{ pkgs ? import {} }: pkgs.mkShell { buildInputs = [ pkgs.python3 pkgs.python3Packages.pip ]; }
    Crea un shell.nix en tu proyecto para definir el entorno:
    </>nix
    { pkgs ? import <nixpkgs> {} }:

pkgs.mkShell {
    buildInputs = [
        pkgs.python3
        pkgs.python3Packages.pip
        pkgs.nodejs
    ];
}

    Luego simplemente ejecuta nix-shell en esa carpeta.

    nix-shell vs venv/pip/venv de Node: nix-shell maneja cualquier dependencia (compiladores, librerías C, herramientas del sistema), no solo paquetes del lenguaje. Es un entorno completo del sistema, no solo de paquetes.

    nix-ld — Binarios externos

    ¿Qué es nix-ld? Permite ejecutar binarios que no fueron compilados para NixOS (como Blender descargado de la web). Proporciona las librerías del sistema que estos programas esperan encontrar en /lib o /usr/lib.

    Configurar nix-ldPermite ejecutar binarios externos (no compilados para NixOS). Añade las librerías que tu app necesite. Ej: Blender → librerías gráficas + CUDA, Steam → steam-libs.Blender: libx11, libglvnd, cuda_cudart
    </>nix
    programs.nix-ld = {
    enable = true;
    libraries = with pkgs; [
        libx11 libxext libxrender
        libglvnd fontconfig freetype
        libjpeg_turbo libpng zlib
        libxkbcommon wayland
        # Para Blender con CUDA:
        linuxPackages.nvidia_x11
        cudaPackages.cuda_cudart
    ];
};

    ¿Cuándo usar nix-ld? Cuando descargas un binario precompilado de su web oficial (Blender, OBS, Spotify, Discord, Steam) y no hay una versión en nixpkgs o necesitas la última versión.

    Wrappers con mkWrapper

    Wrappers son "envoltorios" que ejecutan un binario con las librerías de Nix. Es más flexible que nix-ld porque puedes tener wrappers distintos por aplicación.

    Crear un wrapper con writeShellScriptBinEnvuelve un binario para que use las librerías de Nix. Útil para apps que necesitas aislar.
    </>nix
    pkgs.writeShellScriptBin "mi-app" ''
    export LD_LIBRARY_PATH=${pkgs.libgl}/lib:$LD_LIBRARY_PATH
    exec /ruta/a/binario "$@"
  ''

    ¿nix-ld vs mkWrapper? nix-ld es global (todos los binarios lo usan). mkWrapper es por-app. Para apps que descargas de internet y quieres que funcione "solo", usa nix-ld. Para personalización fina, usa wrappers.

    🖥️ Hardware y Drivers

    Configurar hardware en NixOS es 100% declarativo. Todo va en configuration.nix: drivers de GPU, virtualización, audio, fuentes, y apps que necesitan librerías del sistema.

    NVIDIA — GPU dual (Prime)

    hardware.opengl ▶ hardware.graphics: Desde NixOS 24.11 (diciembre 2024), hardware.opengl fue renombrado a hardware.graphics. Si tu config aún usa opengl, cámbialo.

    ✓ Recomendado
    Configuración NVIDIA con PrimePara portátiles con Intel + NVIDIA. Sync: ambas GPUs siempre activas (máx rendimiento, mayor consumo). Offload: solo la NVIDIA se activa cuando la necesitas (ahorro de batería). Necesitas los Bus IDs de tus GPUs.Sync: gaming/escritorio. Offload: portátil con batería.
    </>nix
    # OpenGL
hardware.graphics = {
    enable = true;
    enable32Bit = true;
};

# NVIDIA driver
services.xserver.videoDrivers = [ "nvidia" ];
hardware.nvidia = {
    modesetting.enable = true;
    open = false;
    package = config.boot.kernelPackages.nvidiaPackages.stable;
    prime = {
        sync.enable = true;
        intelBusId = "PCI:5:0:0";
        nvidiaBusId = "PCI:1:0:0";
    };
};

    Sync vs Offload: sync = rendimiento máximoAmbas GPUs siempre activas. Ideal para escritorio o si no te importa la batería. — para portátiles con batería, usa offload.enable = true y lanza apps con __NV_PRIME_RENDER_OFFLOAD=1.

    Cómo obtener tus Bus IDs: Ejecuta lspci | grep -E "VGA|3D" para ver tus GPUs. Luego convierte los números de hexadecimal a decimal (ej: 01:00.0 → PCI:1:0:0, 05:00.0 → PCI:5:0:0).

    nix-ld — Apps externas (Blender, Steam, etc.)

    ¿Qué es nix-ld? Permite ejecutar binarios que no fueron compilados para NixOS (como Blender descargado de la web). Proporciona las librerías del sistema que estos programas esperan encontrar en /lib o /usr/lib.

    Configurar nix-ldPermite ejecutar binarios externos (no compilados para NixOS). Añade las librerías que tu app necesite. Ej: Blender → librerías gráficas + CUDA, Steam → steam-libs.Blender: libx11, libglvnd, cuda_cudart
    </>nix

    enable = true;
    libraries = with pkgs; [
        libx11 libxext libxrender
        libglvnd fontconfig freetype
        libjpeg_turbo libpng zlib
        libxkbcommon wayland
        # Para Blender con CUDA:
        linuxPackages.nvidia_x11
        cudaPackages.cuda_cudart
    ];
};

    ¿Cuándo usar nix-ld? Cuando descargas un binario precompilado de su web oficial (Blender, OBS, Spotify, Discord, Steam) y no hay una versión en nixpkgs o necesitas la última versión.

    Blender descargado (binario precompilado)

    Blender descargado de la web requiere configuración extra para Vulkan y OpenGL en NixOS. Se usa un wrapper bash + un archivo .desktop para el launcher.

    Launcher scriptScript que configura las variables de entorno necesarias para GPU Nvidia y ejecuta Blender.
    Guardar en: ~/.scripts/blender_launcher.sh
    </>bash
    #!/usr/bin/env bash
# Blender launcher script for NixOS (Reforzado)
BLENDER_PATH="$1"
shift

if [[ -z "$BLENDER_PATH" ]]; then
    echo "Error: No blender path provided"
    exit 1
fi

BLENDER_DIR="$(dirname "$BLENDER_PATH")"

# Soporte para NixOS: Búsqueda dinámica de librerías críticas en el Store
_find_lib64() {
    local libname="$1" pattern="$2"
    for dir in /nix/store/$pattern/lib; do
        [ -f "$dir/$libname" ] && echo "$dir" && return
    done
}

# Buscar glib y dbus (vitales para PySide6/Qt en NixOS)
export LD_LIBRARY_PATH=$(_find_lib64 "libglib-2.0.so.0" "*-glib-[0-9]*"):$LD_LIBRARY_PATH
export LD_LIBRARY_PATH=$(_find_lib64 "libdbus-1.so.3" "*-dbus-[0-9]*-lib"):$LD_LIBRARY_PATH

# Soporte para nix-ld y Drivers
[ -n "$NIX_LD_LIBRARY_PATH" ] && export LD_LIBRARY_PATH="$NIX_LD_LIBRARY_PATH:$LD_LIBRARY_PATH"
export VK_ICD_FILENAMES=/run/opengl-driver/share/vulkan/icd.d/nvidia_icd.x86_64.json
export LD_LIBRARY_PATH=/run/opengl-driver/lib:$LD_LIBRARY_PATH

exec "$BLENDER_PATH" "$@"
    Archivo .desktopArchivo para que Plasma/KDE reconozca Blender como aplicación.
    Guardar en: ~/.local/share/applications/blender-X.X.X.desktop
    </>desktop
    [Desktop Entry]
Name=Blender 5.1.0
Exec=/home/zebus3d/.scripts/blender_launcher.sh /home/zebus3d/Downloads/Blenders/blender-5.1.0-linux-x64/blender %f
Icon=/home/zebus3d/.icons/custom/blender.svg

Categories=Graphics;3DGraphics;

    ⚠️ Ojo con el Garbage Collector: Al ejecutar nix-collect-garbage, se borran librerías del store que programas que usan PySide6 necesitan. Si obtienes errores como libglib-2.0.so.0 not found, es porque tu lanzador no está inyectando estas librerías desde el store.

    ¿Por qué el lanzador reforzado? Aunque Blender traiga sus librerías, los procesos hijos (como un editor de scripts externo) heredan el entorno de Blender. El script reforzado asegura un "pasillo seguro" para estas librerías incluso tras limpiezas profundas del sistema.

    💡 Tip para Addons: Para evitar errores de segmentación (SIGSEGV), filtra el .venv de tu proyecto del sys.path antes de lanzar procesos que usen Qt. Esto evita que se carguen dos versiones distintas de la librería a la vez.
